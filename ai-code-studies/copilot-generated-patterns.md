# Copilot-Generated Code Patterns

*Initial observations — qualitative. Quantitative analysis with LRS benchmarks in progress.*

---

## Background

GitHub Copilot has been in broad production use since 2022. By late 2024, GitHub reported that Copilot was responsible for 40-55% of new code in repositories where it was enabled. We now have enough history to observe patterns in what that code looks like structurally.

These are observations from analyzing OSS repositories where Copilot use is documented or inferable. They are not conclusions — they are hypotheses to test with data.

---

## Pattern 1: Utility Function Proliferation

**Observation:** Copilot tends to generate new utility functions rather than reusing existing ones, because it optimizes for the local context (current file, current prompt) rather than global codebase knowledge.

**Effect:** Codebases accumulate near-duplicate utility functions across modules. Standard linters don't flag this. Hotspots surfaces it as increased coupling fan-out and rising change frequency in utility layers.

**Example pattern:**
```
src/utils/format.ts       — formats dates
src/components/formatDate.ts  — also formats dates, slightly differently
src/api/dateHelpers.ts    — also formats dates, from a third direction
```

None of these is wrong in isolation. Together, they create three change surfaces where one would do.

---

## Pattern 2: Verbose Error Handling

**Observation:** Copilot-generated code tends toward explicit, verbose error handling — often copy-pasting similar try/catch blocks rather than centralizing error logic.

**Effect:** Error handling code becomes a significant portion of the codebase with high duplication. When error handling behavior needs to change, the change is non-obvious and often incomplete.

This is a classic case where the code looks diligent (lots of error handling!) but is structurally fragile (errors handled 40 different ways).

---

## Pattern 3: Test Generation Gaps

**Observation:** When Copilot generates tests, it tends to test the happy path thoroughly and generate edge case tests that are structurally similar to the happy path rather than adversarial.

**Effect:** Code coverage metrics look healthy. Actual branch coverage of error paths and edge cases is lower than it appears.

This is dangerous because it creates false confidence in test suites — a metric that humans and CI systems both rely on.

---

## Pattern 4: Increased Apparent Cohesion, Decreased Actual Cohesion

**Observation:** Copilot-generated modules often appear well-organized — they follow the naming conventions of the surrounding code and have reasonable function names. But the internal logic frequently has hidden dependencies that aren't expressed in the interface.

**Effect:** Code that looks modular isn't. Refactoring a "clean" Copilot-generated module turns out to require touching more of the codebase than expected.

This is one of the hardest patterns to detect without running structural analysis.

---

## What We're Measuring

To quantify these patterns, we're running Hotspots LRS against:

- [ ] A sample of OSS repos before Copilot adoption vs. after (using commit dates and self-reported Copilot usage)
- [ ] PRs explicitly labeled "generated by Copilot" vs. human-authored PRs in the same repo
- [ ] Repositories that have subsequently had to refactor "AI-generated" sections

Results will be published in `code-complexity-dataset/` when the analysis pipeline is running.
